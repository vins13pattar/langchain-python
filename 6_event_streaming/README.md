# 6_event_streaming — LangChain Event Streaming Examples

> **Stream Real-Time Updates from LangChain Agent Runs**
>
> LangChain implements an Event Streaming system powered by LangGraph. In modern versions (v1.3+ / protocol version `"v3"`), **Event Streaming** is the recommended way to stream outputs. It returns a run object with **typed projections** so you can consume independent streams (messages, values, tool calls, subgraphs) without complex state branching.

## Projections at a Glance

The Event Streaming API yields a **run object** with specific, named channels called **projections**:

| Projection | Type | Yields | Use Case |
|------------|------|--------|----------|
| `stream.messages` | Iterator | `ChatModelStream` | Real-time chatbot token and tool-call argument streaming |
| `stream.tool_calls` | Iterator | `ToolCallStream` | Tracking when tools start, stream output, or finish/error |
| `stream.values` | Iterator | `StateSnapshot` | Inspecting state data updates after each node in the graph finishes |
| `stream.output` | State Dict | final agent state | Retrieving the final state output of the entire graph |
| `stream.subgraphs` | Iterator | Subgraph streams | Consuming streaming events from child agents / sub-graphs |

## Core Patterns

### 1 — Token Streaming

Stream output characters to the user live as they are being generated by the model.

```python
stream = agent.stream_events(INPUT, version="v3")

# Loop through messages stream
for message in stream.messages:
    # Loop through text tokens generated by the LLM
    for delta in message.text:
        print(delta, end="", flush=True)
```

---

### 2 — Tool-Call Argument Streaming

See which arguments the model is compiling *before* the tool actually begins executing. This is highly useful for displaying typing indicators or loading states.

```python
stream = agent.stream_events(INPUT, version="v3")

for message in stream.messages:
    # Stream tool argument fragments as the model thinks
    for chunk in message.tool_calls:
        if chunk.get("name"):
            print(f"Tool starting: {chunk['name']}")
        if chunk.get("args"):
            print(f"Arg chunk: {chunk['args']}")
```

---

### 3 — Tool Execution Streaming

Monitor the actual execution lifecycle of tools, including their inputs, output updates, final returns, or errors.

```python
stream = agent.stream_events(INPUT, version="v3")

# Drain messages silently to focus on tool execution
for call in stream.tool_calls:
    print(f"Executing: {call.tool_name} with input {call.input}")
    
    # Drain tool output deltas
    for delta in call.output_deltas:
        pass
        
    if call.error:
        print(f"❌ Failed: {call.error}")
    else:
        print(f"✅ Succeeded: {call.output}")
```

---

### 4 — State Values Streaming

Observe state variables (such as message list additions, custom count tracking, or user names) after each individual step.

```python
stream = agent.stream_events(INPUT, version="v3")

for state in stream.values:
    # state is a snapshot of the graph variables at that point
    current_messages = state["messages"]
    print(f"Current message count: {len(current_messages)}")
```

---

### 5 — Async Streaming

Always use async methods in production environments (like web servers or fast API layers) to prevent thread blocking.

```python
import asyncio

async def run_stream():
    stream = await agent.astream_events(INPUT, version="v3")
    async for message in stream.messages:
        async for delta in message.text:
            print(delta, end="", flush=True)

asyncio.run(run_stream())
```

## Key Rules

1. **Always set `version="v3"`** — Older versions (`v1` or `v2`) use the raw streaming events format and lack the clean, named projection channels.
2. **Use `str(message.text)` for convenience** — If you want the final accumulated string from a message without printing token fragments, drain the deltas first, then convert the `message.text` wrapper directly to a string.
3. **Drain Projections Completely** — If you access `stream.output` (the final state) or `stream.values`, you must fully iterate/drain `stream.messages` (or the stream itself) first to let the execution finish.
4. **Tool IDs match** — Parallel tool runs will output multiple entries inside `stream.tool_calls`. You can distinguish them using their unique `call.id`.
